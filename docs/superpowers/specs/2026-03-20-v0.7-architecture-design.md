# Caboose 0.7 / 0.8 Architecture вҖ” Design Spec

**Date:** 2026-03-20
**Scope:** Three-layer architecture extraction, background agents, WebSocket server, mobile app
**Repos:** `kahanscious/caboose` (core + TUI), `kahanscious/caboose-mobile` (Flutter app, private)

---

## Overview

Caboose 0.7 is an architectural milestone. The monolithic TUI binary is decomposed into three layers: a platform-agnostic core, an orchestration layer, and a thin TUI frontend. This enables background agents, a WebSocket remote API, and a mobile app вҖ” all consuming the same core.

### Release Cadence

| Release | Scope | Description |
|---------|-------|-------------|
| **0.7.0** | Core extraction + app.rs breakup | Three-layer architecture. No new features, no behavior changes. All existing tests pass. |
| **0.7.x** | Background agents + WebSocket server | `/bg` command, general-purpose remote API over WebSocket |
| **0.8.0** | Mobile app | Flutter Android native + Flutter Web PWA (iOS). Session continuity, GitHub/GitLab integration |

## 1. Three-Layer Architecture

### The Problem

`app.rs` is 13.6K lines containing domain logic, orchestration, and rendering in one struct. The `State` struct has 300+ fields mixing agent state, session data, provider config, UI scroll positions, and dialog state. This makes it impossible to reuse the core logic from mobile or headless contexts.

### Layer Responsibilities

**`caboose-core`** (new Rust crate вҖ” `core/` in workspace):
- Agent state machine (conversation, streaming, tool call decisions, compaction triggers)
- Provider trait, registry, pricing
- Session types, persistence (SQLite), migrations (`migrate/`)
- Tool definitions and execution (shell, file I/O, git)
- Config schema and loading
- Memory system (observations, search, extraction)
- Checkpoint/rewind logic
- MCP client
- Hooks lifecycle (fire events, parse responses)
- Skills (loading, resolution, builtins) вҖ” `skills/`
- Circuits (scheduled tasks, runner) вҖ” `circuits/`
- SCM integration (detection, tools, watcher) вҖ” `scm/`
- Safety (command policy, env filter) вҖ” `safety/`
- Suggest (scanner, parsers, priority) вҖ” `suggest/`
- Sub-agent infrastructure (conflict detection, executor, worktree, pipeline) вҖ” `sub_agent/`
- Roundhouse engine (planner, session, types) вҖ” `roundhouse/`
- Init generation (`init/`)
- Attachment handling (`attachment.rs`)
- Event-driven interface: emits `CoreEvent`, accepts `CoreCommand`

**Modules that stay in TUI** (desktop-specific):
- LSP client (`lsp/`) вҖ” requires local filesystem + running language servers. Desktop-only, not meaningful for mobile.
- Terminal panel (`terminal/`) вҖ” embedded PTY, inherently TUI-specific.
- Clipboard (`clipboard.rs`) вҖ” platform-specific copy/paste.
- Update checker (`update.rs`) вҖ” desktop binary update mechanism.

**`caboose-tui`** (existing `tui/` crate вҖ” slimmed down):
- Terminal rendering (ratatui widgets, layout, themes)
- Key/mouse input handling
- View models (maps `CoreEvent` to renderable state)
- Dialog management (approval prompts, pickers, inputs)
- Slash command UI (autocomplete, inline dropdowns)
- Roundhouse TUI (split panels, phase navigation)

**`caboose-mobile`** (separate repo вҖ” Flutter):
- Flutter UI (Android native + Web PWA)
- Maps `CoreEvent` to Flutter widgets
- WebSocket client for desktop connection
- GitHub/GitLab OAuth + API client
- On-device API key storage

## 2. Event Contract: `CoreEvent` and `CoreCommand`

The core communicates with consumers exclusively through typed events. No consumer calls core methods directly except to send commands.

### `CoreEvent` вҖ” Core to Consumer (outbound)

```rust
pub enum CoreEvent {
    // --- Streaming ---
    TextDelta(String),
    ThinkingDelta(String),
    StreamStarted { model: String },

    // --- Tool lifecycle ---

    SessionCreated { id: String },
```

### Event Bus

Core exposes a simple interface:

```rust
pub struct CabooseCore {
    event_tx: broadcast::Sender<CoreEvent>,
    command_rx: mpsc::UnboundedReceiver<CoreCommand>,
}

impl CabooseCore {
    pub fn new(config: Config) -> (Self, CoreHandle);
    pub async fn run(&mut self); // main event loop
}

pub struct CoreHandle {
    pub events: broadcast::Receiver<CoreEvent>,
    pub commands: mpsc::UnboundedSender<CoreCommand>,
}
```

Multiple consumers can subscribe to events (broadcast channel). Commands are funneled through a single `mpsc` channel (multiple senders, single receiver вҖ” core). The TUI gets a `CoreHandle`. The WebSocket server gets a `CoreHandle`. Background agents each get their own `CabooseCore` instance (separate event bus).

**Tool approval ownership:** `ToolApprovalRequest` events go to all subscribers via broadcast, but only the **primary consumer** should respond. The core tracks a `primary_consumer_id` вҖ” set to TUI by default, switchable to a mobile client during session handoff. If a non-primary consumer sends `ApproveTool`, core ignores it. If the primary disconnects (mobile goes offline), ownership reverts to TUI. This prevents duplicate approvals when both TUI and mobile are connected.

**Approval timeout:** If no response arrives within 60 seconds (configurable), core emits `ApprovalTimeout` and the agent stalls until a consumer responds or the user cancels.

## 3. `app.rs` Decomposition

The 13.6K line `app.rs` breaks into focused modules. This happens as part of core extraction вҖ” not as a separate step.

### Current `State` struct вҶ’ split into three:

**`CoreState`** (lives in `caboose-core`):
- `agent: AgentLoop`
- `session_manager: SessionManager`
- `config: Config`
- `pricing: PricingRegistry`
- `checkpoints: CheckpointManager`
- `provider: Option<Box<dyn Provider>>`
- `compaction_provider: Option<Box<dyn Provider>>`
- `active_provider_name: String`
- `active_model_name: String`
- Session cost/token accumulators

**`OrchestrationState`** (lives in `caboose-core`, managed by event loop):
- `tool_exec_queue: VecDeque<PendingToolCall>`
- `tool_exec_results: Vec<ToolResult>`
- `message_queue: VecDeque<QueuedMessage>`
- `discovered_locals: Vec<LocalServer>`
- Background agent registry

**`ViewState`** (lives in `caboose-tui`):
- `scroll_offset`, `sidebar_width`, `sidebar_visible`
- `hovered_message`, `hovered_code_block`
- `expanded_messages`, `expanded_thinking`
- `slash_auto`, `dialog_stack`
- `input`, `paste detection state`
- Chat messages (presentation format, not domain)

### Async Channel Migration

The current `State` struct holds ~9 async receivers for in-flight operations. Each must be classified:

| Channel | Purpose | Destination |
|---------|---------|-------------|
| `init_rx` | CABOOSE.md generation streaming | Core (init is domain logic) |
| `tool_exec_pending_rx` | Pending tool execution results | Core (orchestration) |
| `sub_agent_rx` | Subagent event stream | Core (agent management) |
| `roundhouse_update_rx` | Planner status updates | Core (roundhouse engine) |
| `roundhouse_synthesis_rx` | Synthesis streaming deltas | Core (roundhouse engine) |
| `roundhouse_critique_rx` | Critique phase updates | Core (roundhouse engine) |
| `mcp_connect_rx` | MCP server connection result | Core (MCP client) |
| `workspace_scan_rx` | Background workspace discovery | TUI (UI-triggered scan) |
| `update_check_rx` | Binary update check result | TUI (desktop-only) |
| `title_rx` | Auto-generated session title | Core (session management) |

Core-bound channels become internal to `CabooseCore` and their results surface as `CoreEvent` variants. TUI-bound channels stay in `ViewState`.

## 4. Background Agents

### Design

Background agents reuse the existing subagent infrastructure (worktree isolation, conflict detection) with a new UX:

- `/bg <prompt>` spawns a background agent
- Each background agent gets its own `AgentLoop` + worktree
- Auto-approve mode: no tool approval prompts (configurable)
- Progress shown in TUI statusline: `[bg: 2 running, 1 done]`
- Completion triggers system notification + merge review

### Budget Enforcement

- Per-agent token budget: `max_bg_agent_cost` config (default: $1.00)
- Global session budget sums across all agents (foreground + background)
- Background agent pauses and emits `BudgetExceeded` when limit hit
- User can continue, raise, or cancel via `CoreCommand`

### Failure Handling

- On crash/error: worktree cleaned up automatically, error reported via `BackgroundAgentFailed` event
- On disconnect: agent continues running (it's a local process), reports result when consumer reconnects
- Worktree GC: on session start, scan for orphaned worktrees from crashed agents and clean up

### Implementation

- Built on `axum` + `tokio-tungstenite`
- Subscribes to `CoreEvent` broadcast channel, forwards to connected clients as JSON
- Accepts `CoreCommand` from clients, forwards to core command channel
- Auth: pairing token (short-lived) exchanged for session token (long-lived, revocable)
- Backpressure: buffer TextDelta events, send accumulated text every 100ms to slow clients
- Optional вҖ” disabled by default, enabled via config or `--serve` flag

### Discovery

- **QR code pairing** (primary): desktop shows QR with `wss://<ip>:<port>?token=<pairing_token>`. Mobile scans, exchanges for session token.
- **mDNS** (secondary, LAN only): advertise `_caboose._tcp` with port. Mobile discovers automatically on same network.
- **Manual URL** (fallback): user enters `<ip>:<port>` in mobile app settings.

## 6. Mobile App

### Tech Stack

- **Flutter** вҖ” single codebase for Android native, Web PWA, and iOS native (deferred)
- **`flutter_rust_bridge`** вҖ” FFI to `caboose-core` for Android/iOS native builds
- **Dart HTTP client** вҖ” direct LLM API access for Web builds (no Rust FFI in browser)
- **Service abstraction** вҖ” `ChatService` interface with two implementations: `NativeChatService` (via core FFI) and `WebChatService` (via HTTP)

### Two-Tier Experience

**Connected (desktop running + WebSocket link):**
- Real-time session streaming (see tool calls, text deltas)
- Approve/deny tool calls from mobile
- Session handoff: "send to mobile" / "send to desktop"
- Full agent capabilities (tools execute on desktop machine)
- Push notifications for: tool approval requests, task completion, errors

**Disconnected (standalone):**
- Chat with LLMs using on-device API keys (Dart HTTP client)
- Browse repos, PRs, branches via GitHub/GitLab API
- View CI/CD status and build logs
- Read-only session history (last synced from desktop)

## 7. Session Sync Model

### Connected Mode

When mobile connects to desktop via WebSocket:

1. Desktop sends full session list (id, title, turn_count, cost, updated_at)
2. Mobile requests specific session history on demand (lazy load)
3. Real-time events stream for the active session
4. Session ownership: **desktop is primary**. Mobile can send messages and approvals, but tool execution always happens on desktop. If both type simultaneously, messages are ordered by arrival at core.

### Disconnected Mode

- Mobile stores session metadata locally (SQLite via `sqflite`)
- Last connected sync populates session history
- Standalone sessions created on mobile are mobile-local
- When reconnecting, mobile-originated sessions can be "promoted" to desktop (future enhancement, not v1)

### Session Data Format

Core defines session types that both TUI and mobile understand:

```rust
// In caboose-core
pub struct SessionSummary {
    pub id: String,
    pub title: Option<String>,
    pub model: Option<String>,
    pub provider: Option<String>,
    pub turn_count: u32,
    pub total_cost_usd: f64,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
}

pub struct SessionMessage {
    pub role: String,        // "user", "assistant", "tool"
    pub content: Vec<ContentBlock>,
    pub timestamp: DateTime<Utc>,
}

pub enum ContentBlock {
    Text(String),
    Thinking(String),
    ToolUse { id: String, name: String, arguments: serde_json::Value },
    ToolResult { id: String, output: String, is_error: bool },
    Image { path: String, media_type: String },
}
```

These serialize to JSON for WebSocket transport and are the DTO layer for flutter_rust_bridge FFI.

## 10. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Core extraction breaks all tests | High | Freeze features, single branch, mechanical migration |
| Event model over-engineered | Medium | Design for TUI + bg agents first, add mobile events later |
| `flutter_rust_bridge` FFI friction | Medium | Budget for DTO layer, keep FFI surface small |
| WebSocket auth security | Medium | Short-lived pairing tokens, TLS, session token revocation |
| Session sync conflicts | Low (v1) | Desktop is primary, mobile read-only when connected |
| iOS PWA limitations | Low | Scope PWA to standalone features, native Flutter later |
| Mobile app scope creep | High | Ship minimal Console + Repos tabs first, iterate |

## 11. Out of Scope

- Caboose cloud account / hosted relay server (self-hosted WebSocket only)
- iOS App Store submission (deferred until demand)
- Real-time collaborative editing (desktop is always primary)
- Mobile tool execution (no filesystem access from phone)
- Cross-device session merge (mobile sessions stay mobile-local in v1)
