# v0.7.1 Design: Server, Background Agents, Search Backends

**Date:** 2026-03-20
**Version:** 0.7.1
**Branch:** `feat/0.7.1` (from `main` after 0.7.0 merges)
**Depends on:** 0.7.0 (core extraction)

## Overview

Three features that together unblock the 0.8.0 mobile app:

1. **`caboose-server`** — new workspace crate. Axum + WebSocket server wrapping `caboose-core`. Embeddable in the TUI or runnable standalone.
2. **Background agents** — `/bg <prompt>` spawns independent `AgentLoop` instances with auto-approve and budget enforcement.
3. **Pluggable search backends** — `SearchBackend` trait in core with SearXNG (tender-highball) and Tavily implementations for the `web_search` tool.

## Prerequisites

### Event Bus (CoreEvent/CoreCommand)

The 0.7.0 architecture spec defined a `CoreEvent`/`CoreCommand` event bus and `CabooseCore`/`CoreHandle` types, but these were not implemented — 0.7.0 only completed the crate split and app.rs decomposition. The TUI currently calls methods directly on `AgentLoop` and consumes `AgentEvent` (the only event enum in core today).

Building the event bus is the **first deliverable of 0.7.1**, as the server depends on it. This means:

1. Define `CoreEvent` enum in `core/src/events.rs` — wraps existing `AgentEvent` variants plus new categories (session, MCP, background agents, etc.)
2. Define `CoreCommand` enum — all inbound actions (send message, tool approval, session CRUD, provider switching, etc.)
3. Implement `CabooseCore` struct with `mpsc` command channel and `broadcast` event channel
4. Migrate TUI from direct method calls to command/event pattern
5. Server consumes the same `CoreEvent` broadcast and sends `CoreCommand` through the same channel

**Existing `AgentEvent` variants** (in `core/src/agent/mod.rs`) that map into `CoreEvent`:
- `TextDelta(String)`
- `ThinkingDelta(String)`
- `ToolCall { id, name, arguments }`
- `TurnComplete { input_tokens, output_tokens, cache_read_tokens, cache_creation_tokens }`
- `Error(String)`
- `ProviderError { category, provider, message, hint }`
- `CompactionComplete`

### Session Cost Tracking Migration

The existing `sessions` SQLite table has no cost column. A migration adds `total_cost_usd REAL DEFAULT 0.0` to the sessions table. Cost was tracked in-memory in 0.6.4 but not persisted to the session row.

## 1. `caboose-server` Crate

### 1.1 Crate Structure

New workspace member `server/`. Both a library (for TUI embedding) and a binary (for standalone/headless use).

```
server/
├── Cargo.toml
└── src/
    ├── lib.rs          # start_server() entry point
    ├── main.rs         # standalone binary (CLI args)
    ├── ws/
    │   ├── mod.rs      # WebSocket upgrade handler
    │   ├── session.rs  # per-connection session actor
    │   └── envelope.rs # JSON message envelope types
    ├── auth/
    │   ├── mod.rs      # middleware + pairing flow
    │   ├── pairing.rs  # QR code generation, one-time code logic
    │   └── devices.rs  # device token storage (SQLite)
    ├── bridge.rs       # CoreEvent/CoreCommand <-> WebSocket JSON mapping
    └── state.rs        # shared server state (AppState)
```

### 1.2 Dependencies

- `axum` — HTTP framework (includes WebSocket support via `axum::extract::ws`, which wraps `tokio-tungstenite` internally — no need for a direct `tokio-tungstenite` dependency)
- `caboose-core` — domain logic (path dependency)
- `serde` / `serde_json` — message serialization
- `qrcode` — QR code generation for pairing
- `rand` — pairing code + device token generation

### 1.3 Server Lifecycle

```rust
pub struct ServerConfig {
    pub port: u16,              // default 9090
    pub bind: String,           // default "0.0.0.0"
    pub core_config: Config,    // caboose-core config
}

pub struct ServerHandle {
    // shutdown signal, pairing code generation, device management
}

pub async fn start_server(config: ServerConfig) -> Result<ServerHandle>
```

- Binds to `{bind}:{port}`
- Returns a `ServerHandle` with `shutdown()`, `generate_pairing_code()`, `list_devices()`, `revoke_device()`
- TUI calls `start_server()` on launch
- Standalone binary parses CLI args and calls the same function

### 1.4 WebSocket Protocol

Unified JSON envelope for all messages:

```json
// Client -> Server (command)
{
  "id": "uuid",
  "type": "command",
  "command": "SendMessage",
  "payload": { "text": "explain this function" }
}

// Server -> Client (event)
{
  "id": "uuid",
  "type": "event",
  "event": "TextDelta",
  "payload": { "text": "This function " }
}

// Server -> Client (tool approval request)
{
  "id": "uuid",
  "type": "event",
  "event": "ToolApprovalRequired",
  "payload": { "tool_calls": [...], "index": 0 }
}

// Client -> Server (approval response)
{
  "id": "uuid",
  "type": "command",
  "command": "ToolApproval",
  "payload": { "index": 0, "approved": true }
}
```

The `id` field enables request-response correlation for CRUD operations (e.g., `ListSessions` command returns a `SessionList` event with the same `id`).

Every `CoreCommand` variant maps to a command type. Every `CoreEvent` variant maps to an event type. The server is a transport layer, not a translation layer.

### 1.5 Connection Model

- One WebSocket connection = one active agent session
- Client sends `AttachSession { session_id }` after auth (or `CreateSession` for new)
- Server spawns a per-connection actor that owns the `AgentLoop` for that session
- Multiple devices can connect to different sessions simultaneously
- All connections use plain `ws://` (unencrypted). This is a LAN-first service; `wss://` via reverse proxy is a future enhancement.

### 1.5.1 Disconnect & Reconnection

When a client disconnects (network drop, app backgrounded):

1. Server keeps the agent running for 30s (grace period)
2. Server buffers events during disconnect (bounded: 1000 events or 1MB, whichever is hit first)
3. Reconnecting client sends `Authenticate` (device token) then `AttachSession { session_id }` as normal
4. Server replays buffered events from the disconnect point
5. If the buffer overflowed during disconnect, server sends a `SessionStateStale` event — client must reload full session state via `LoadSession`
6. After 30s with no reconnection, agent tears down and session is saved

### 1.6 Backpressure

`TextDelta` events batch into 50ms windows before sending to avoid flooding slow mobile connections. Client can override via:

```json
{ "type": "command", "command": "SetStreamRate", "payload": { "ms": 100 } }
```

### 1.7 API Surface

Everything that exists in `caboose-core` is exposed. Organized by category, split into commands/events that map to existing core code vs. new API surface that 0.7.1 introduces.

#### Existing core functionality (direct mapping to current code)

**Conversation** (maps to existing `AgentLoop` methods + `AgentEvent` variants):
- `SendMessage` command → `AgentLoop::send_message()` (starts a turn)
- `CancelTurn` command → cancels the active provider stream
- `TextDelta` event ← `AgentEvent::TextDelta`
- `ThinkingDelta` event ← `AgentEvent::ThinkingDelta`
- `ToolCall` event ← `AgentEvent::ToolCall`
- `TurnComplete` event ← `AgentEvent::TurnComplete`
- `Error` event ← `AgentEvent::Error`
- `ProviderError` event ← `AgentEvent::ProviderError`
- `CompactionComplete` event ← `AgentEvent::CompactionComplete`

**Tool approval** (maps to existing `AgentState::PendingApproval` + `check_permission()`):
- `ToolApprovalRequired` event ← when state enters `PendingApproval`
- `ToolApproval` command → approve/deny per tool, resumes execution
- `ToolResult` event ← `ToolResult` struct from `execute_tool()`

**Sessions** (maps to existing `SessionManager` methods):
- `CreateSession` → `SessionManager::create()`
- `ListSessions` → `SessionManager::list_with_content()`
- `LoadSession` → `SessionManager::get()` + `load_messages()`
- `DeleteSession` → `SessionManager::delete()`
- `SearchSessions` → `SessionManager::search()`
- Response events: `SessionCreated`, `SessionList`, `SessionLoaded`, `SessionDeleted`, `SessionSearchResults`

**Providers** (maps to existing `ProviderRegistry` + `Provider` trait):
- `ListProviders` → `ProviderRegistry::list_providers()`
- `SwitchProvider` → construct new provider via registry
- `ListModels` → `Provider::list_models()`
- `SwitchModel` → set model on current provider
- `SetThinkingMode` → `Provider::set_thinking_mode()`
- Response events: `ProviderList`, `ProviderSwitched`, `ModelList`, `ModelSwitched`, `ThinkingModeChanged`

**MCP** (maps to existing `McpManager`):
- `ListMcpServers` → `McpManager::servers` state
- `ConnectMcpServer` → `McpManager::connect()`
- `DisconnectMcpServer` → `McpManager::disconnect()`
- Response events: `McpServerList`, `McpServerConnected`, `McpServerDisconnected`, `McpToolsDiscovered`

**Skills** (maps to existing skills module):
- `ListSkills` → skill resolution from config + builtins
- `InvokeSkill` → inject skill template into conversation
- Response events: `SkillList`, `SkillInvoked`

**Checkpoints** (maps to existing `checkpoint` module):
- `CreateCheckpoint`, `ListCheckpoints`, `RewindToCheckpoint`
- Response events: `CheckpointCreated`, `CheckpointList`, `CheckpointRewound`

**Roundhouse** (maps to existing `RoundhouseSession`):
- `StartRoundhouse`, `SetRoundhouseProviders`
- Events: `RoundhousePhaseChanged`, `RoundhousePlannerUpdate`, `RoundhouseComplete`

**Sub-agents** (maps to existing `sub_agent` module):
- `SubAgentSpawned`, `SubAgentUpdate`, `SubAgentComplete` events (observation only — spawned by the agent, not by the user)

**SCM** (maps to existing `scm` module):
- `GetScmStatus` → SCM detection + status
- Response event: `ScmStatus`

#### New API surface (introduced in 0.7.1)

**Background agents** (see section 3):
- `SpawnBackgroundAgent`, `ListBackgroundAgents`, `AttachBackgroundAgent`, `DetachBackgroundAgent`, `KillBackgroundAgent`
- Events: `BackgroundAgentStarted`, `BackgroundAgentProgress`, `BackgroundAgentComplete`, `BackgroundAgentFailed`

**Server-level commands** (do not flow through core):
- `SetStreamRate { ms }` — backpressure tuning (see section 1.6)
- `GetStatus` → `Status` event (current model, provider, permission mode, session info, server uptime, connected devices count)

## 2. Device Pairing Auth

### 2.1 Pairing Flow

```
1. User runs /pair in TUI (or server starts in standalone mode)
2. Server generates one-time pairing code (6-char alphanumeric, 5-minute expiry)
3. TUI displays QR code: { "url": "ws://192.168.1.x:9090", "code": "A3F8K2" }
4. Mobile scans QR, opens WebSocket, sends:
   { "type": "auth", "command": "Pair", "payload": { "code": "A3F8K2", "device_name": "Alex's Pixel" } }
5. Server validates code (one-time, not expired), burns it
6. Server generates device token (256-bit random), stores device in SQLite
7. Server responds:
   { "type": "auth", "event": "Paired", "payload": { "device_token": "...", "device_id": "uuid" } }
8. Mobile stores device token locally, uses for all future connections
```

### 2.2 Reconnection

```
1. Mobile opens WebSocket, sends:
   { "type": "auth", "command": "Authenticate", "payload": { "device_token": "..." } }
2. Server validates token against devices table
3. Server responds:
   { "type": "auth", "event": "Authenticated", "payload": { "device_id": "...", "device_name": "..." } }
4. Connection is live — client can send commands
```

### 2.3 Device Management

- `/pair` — generate new pairing code + QR display
- `/devices` — list paired devices (name, last seen, paired date)
- `/unpair <device_id>` — revoke device token, close active connections from that device

### 2.4 Storage

New `devices` table in the existing sessions SQLite database:

```sql
CREATE TABLE devices (
    id            TEXT PRIMARY KEY,
    name          TEXT NOT NULL,
    token_prefix  TEXT NOT NULL,      -- first 8 bytes of token (hex), indexed for fast lookup
    token_hash    TEXT NOT NULL,      -- argon2 hash of full token, never store raw
    paired_at     TEXT NOT NULL,
    last_seen     TEXT,
    revoked       INTEGER DEFAULT 0
);

CREATE INDEX idx_devices_token_prefix ON devices(token_prefix) WHERE revoked = 0;
```

**Lookup flow:** Client sends full device token. Server extracts the first 8 bytes as prefix, queries `WHERE token_prefix = ? AND revoked = 0` (indexed, instant), then verifies the full token against `token_hash` with argon2 for the matched row(s).

**Max devices:** 10 per server instance. Attempting to pair an 11th device returns an error with instructions to `/unpair` an existing device first.

### 2.5 Security

- All messages before auth are rejected. Server sends `AuthRequired` and closes after 10s.
- Pairing codes are single-use and expire after 5 minutes.
- Device tokens are long-lived but individually revocable.
- Provider API keys never transit the WebSocket — they stay on the server.
- Rate limit: max 5 failed auth attempts per IP per minute.

## 3. Background Agents

### 3.1 Core Model

A background agent is an independent `AgentLoop` on its own Tokio task with `Chug` permission mode (auto-approve all tools).

### 3.2 Commands

```
/bg <prompt>              — spawn a new background agent
/bg list                  — show running/completed agents
/bg attach <id>           — stream background agent output to foreground
/bg detach                — return to main session
/bg kill <id>             — cancel a running agent
/bg budget <tokens>       — set default per-agent budget
```

### 3.3 Budget Enforcement

Two levels:

- **Per-agent budget** — default 100K tokens (configurable). Agent killed when exceeded.
- **Global budget** — total across all concurrent background agents. Prevents runaway spend.

Budget checked after each turn. When approaching the limit, agent receives a system message: "You are approaching your token budget. Wrap up your current task." Hard-killed after that turn completes.

### 3.4 Lifecycle

```
Spawned -> Running -> Completed | Killed | BudgetExceeded | Error
```

- Each background agent gets its own session in SQLite (`background: true`, linked via `parent_session_id`)
- Status updates emitted as granular `CoreEvent` variants:
  ```rust
  // Agent spawned and running
  BackgroundAgentStarted {
      id: String,
      prompt_summary: String,    // first ~50 chars of prompt
      budget: u64,               // per-agent token budget
      parent_session_id: String,
  }

  // Periodic progress (emitted after each turn)
  BackgroundAgentProgress {
      id: String,
      tokens_used: u64,
      budget_remaining: u64,
      turn_count: u32,
  }

  // Agent finished its task successfully
  BackgroundAgentComplete {
      id: String,
      tokens_used: u64,
      session_id: String,        // session ID for reviewing output
  }

  // Agent failed (error, budget exceeded, or killed)
  BackgroundAgentFailed {
      id: String,
      reason: BackgroundAgentFailReason,  // Error(String), BudgetExceeded, Killed
      tokens_used: u64,
  }
  ```
- TUI shows `[bg: 2 running]` in the statusline
- Mobile receives the same events over WebSocket

### 3.5 Attach Mode

`/bg attach <id>` switches the foreground view to stream that agent's output in real-time. Read-only — the agent continues auto-approving. `/bg detach` returns to the main session.

### 3.6 Concurrency

No artificial limit beyond a configurable `max_concurrent` safety cap (default 5). Each agent runs in its own Tokio task with its own `AgentLoop` and provider instance. The global budget is the natural throttle.

## 4. Pluggable Search Backends

### 4.1 Trait

In `core/src/tools/search.rs`:

```rust
#[async_trait]
pub trait SearchBackend: Send + Sync {
    async fn search(&self, query: &str, max_results: usize) -> Result<Vec<SearchResult>>;
    fn name(&self) -> &str;
}

pub struct SearchResult {
    pub title: String,
    pub url: String,
    pub snippet: String,
}
```

### 4.2 SearXNG Backend (Tender Highball)

`core/src/tools/search_searxng.rs`:

- `GET {base_url}/search?q={query}&format=json&language=en`
- Custom `User-Agent` header matching the Cloudflare WAF rule
- 5s request timeout
- Maps SearXNG `results[]` array to `Vec<SearchResult>`
- Handles partial results gracefully (some upstream engines may fail)

### 4.3 Tavily Backend

`core/src/tools/search_tavily.rs`:

- `POST https://api.tavily.com/search`
- API key from `AuthStore` (stored like any other provider key)
- Standard Tavily response mapping

### 4.4 Configuration

The existing `ServicesConfig` uses a flattened `HashMap<String, ServiceConfig>` where each service has `provider`, `api_key_env`, and `enabled` fields. We extend `ServiceConfig` with optional fields rather than introducing a new schema shape:

```rust
pub struct ServiceConfig {
    pub provider: String,                    // "tavily" or "searxng"
    pub api_key_env: Option<String>,         // for tavily: "TAVILY_API_KEY"
    pub enabled: bool,                       // default true
    pub base_url: Option<String>,            // for searxng: "https://search.trycaboose.dev"
    pub user_agent: Option<String>,          // for searxng: "Caboose/1.0 (SearchService)"
    pub max_results: Option<usize>,          // default 10
}
```

**SearXNG config:**
```toml
[services.web_search]
provider = "searxng"
base_url = "https://search.trycaboose.dev"
user_agent = "Caboose/1.0 (SearchService)"
max_results = 10
```

**Tavily config** (existing pattern, unchanged):
```toml
[services.web_search]
provider = "tavily"
api_key_env = "TAVILY_API_KEY"
```

**No config:** `web_search` tool returns a helpful error with setup instructions.

### 4.5 Tool Execution Flow

1. `execute_tool("web_search", ...)` is called
2. Reads `config.services["web_search"]` to resolve backend via the `provider` field
3. Constructs appropriate `SearchBackend` implementation
4. Calls `backend.search(query, max_results)`
5. Formats results as tool output string for the LLM
6. If no backend configured, returns helpful error with setup instructions

### 4.6 Testing Strategy

- **Unit tests:** Mock HTTP server (`wiremock`) returning SearXNG-shaped JSON. Tests parsing, error handling, timeouts, partial results.
- **Integration fixtures:** Real response shapes from tender-highball docs used as test fixtures.
- **Config tests:** Backend resolution from TOML, fallback when unconfigured, invalid backend name errors.
- **Tavily tests:** Same pattern with Tavily response fixtures.

## 5. TUI Integration

### 5.1 Server Embedding

- `App::new()` calls `start_server(config)` on startup if `config.server.enabled`
- Stores `ServerHandle` in `App` state
- `/pair` triggers `server_handle.generate_pairing_code()`, renders QR in a TUI dialog
- `/devices` and `/unpair` are new slash commands calling through the handle
- Server shuts down on TUI exit via `server_handle.shutdown()`

### 5.2 Background Agent Commands

- `/bg` family added to `app/slash_commands.rs`
- Status indicator in TUI statusline
- `/bg attach` reuses conversation view, switching event source
- `BackgroundAgentStarted/Progress/Complete/Failed` events handled in event loop alongside normal events

### 5.3 Configuration

```toml
[server]
enabled = true          # auto-start server with TUI (default: true)
port = 9090
bind = "0.0.0.0"

[background_agents]
per_agent_budget = 100000    # tokens
global_budget = 500000       # total across concurrent agents
max_concurrent = 5

[services.web_search]
provider = "searxng"
base_url = "https://search.trycaboose.dev"
user_agent = "Caboose/1.0 (SearchService)"
```

## Non-Goals for 0.7.1

- Remote access beyond LAN (Cloudflare Tunnel for server — future)
- Web UI client (mobile app is the client, 0.8.0)
- End-to-end encryption on WebSocket (LAN trust model for now)
- Multi-user support (single user, multiple devices)
