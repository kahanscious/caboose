## Feature 1: Background Agent Wiring

### 1.1 Entry Points

Two ways to spawn a background agent:

1. **`/bg <prompt>`** â€” explicit user command. Spawns immediately with the active provider/model unless `--model <model>` is specified.
2. **`spawn_background` tool** â€” LLM-initiated. The agent decides when a task should run in the background, same pattern as the existing `spawn_agent` tool for subagents. The LLM receives system prompt guidance on when to use it.

### 1.2 System Prompt Guidance

Added to the existing subagents block in the system prompt:

> You also have a `spawn_background` tool. Use it for tasks that are independent of the current conversation and don't need interactive feedback â€” long test runs, large refactors across many files, code generation tasks where the user doesn't need to watch progress. Don't use it for tasks where the user is waiting for the answer. Background agents run with auto-approve and report results when done.

### 1.3 Spawn Flow

1. Check `BackgroundAgentManager::can_spawn()` â€” respects `max_concurrent` and `global_budget` limits.
2. If denied, post system message explaining why (budget exceeded or concurrency limit).
3. Create a new `AgentLoop` with:
   - The specified (or active) provider/model
   - Auto-approve permission mode (Chug â€” no tool approval prompts)
   - The user's prompt as the initial message
   - A unique agent ID (UUID)
   - **Tool registry:** Full tool set minus `spawn_agent` and `spawn_background` (prevents recursive spawning). Same restriction pattern as the subagent executor (`tui/src/sub_agent/executor.rs`).
   - **MCP:** No MCP connections. Background agents run with built-in tools only. Adding MCP would double server connections per agent.
   - **System prompt:** Minimal â€” the base system prompt without memory/skills/agent awareness blocks. Background agents are task runners, not conversational.
   - **Working directory:** Same working directory as the main session (no isolated worktree). Background agents should avoid writing to files the user is actively editing. The system prompt tells the agent to be cautious about file writes. If concurrent write conflicts become a problem, worktree isolation can be added later.
   - **Session ID:** New UUID for the background agent's own conversation. Parent session ID is `state.current_session_id`.
4. Call `BackgroundAgentManager::register()` with `session_id` and `parent_session_id` â€” emits `CoreEvent::BackgroundAgentStarted`.
5. Spawn on a tokio task. Store the `JoinHandle` via `store_handle()`.
6. The spawned task:
   - Runs the agent loop to completion
   - Calls `update_tokens()` periodically
   - On success: calls `mark_complete()`, emits `BackgroundAgentComplete`
   - On failure: calls `mark_failed()`, emits `BackgroundAgentFailed`

### 1.4 CoreEvent Consumption

The TUI event loop (`tui/src/app/mod.rs`) does not currently consume `CoreEvent`. To receive background agent lifecycle events:

1. On `App::new()`, create a `CoreHandle` and store the `broadcast::Receiver` in `State` as `core_event_rx: Option<tokio::sync::broadcast::Receiver<CoreEvent>>`.
2. Pass the `CoreHandle` clone to `BackgroundAgentManager::new()`.
3. In the main event loop (`App::run()`), poll `core_event_rx.try_recv()` each tick (same pattern as `sub_agent_rx`).
4. Handle relevant variants:
   - `BackgroundAgentStarted { id, prompt }` â€” add to sidebar agents list
   - `BackgroundAgentComplete { id, tokens_used }` â€” update sidebar status, post system message
   - `BackgroundAgentFailed { id, reason }` â€” update sidebar status, post error message
   - All other variants â€” ignore (they're for the server/mobile, not TUI)

### 1.5 Provider/Model Resolution

`/bg` defaults to the active provider and model in the current session. The `--model` flag accepts any model ID that Caboose supports. Resolution follows the same cascade as the main session:

- If the model ID matches a known provider (e.g. `claude-haiku-4-5` â†’ anthropic), use that provider's API key.
- If the model ID is an OpenRouter model path (e.g. `anthropic/claude-haiku-4-5`), route through OpenRouter.
- If the model ID doesn't resolve, post an error message.

The `spawn_background` tool accepts an optional `model` parameter with the same resolution logic.

**`--model` flag parsing:** Simple manual parsing â€” check if `args` starts with `--model`, consume the next token as the model ID, and treat the remainder as the prompt. No arg parser dependency needed. Example: `/bg --model claude-haiku-4-5 run all tests` â†’ model=`claude-haiku-4-5`, prompt=`run all tests`.

### 1.5 Notification

- **Sidebar:** The existing agents section in the sidebar shows background agents with status (Running / Complete / Failed) and token usage. Reuses the same rendering pattern as subagents.
- **System message:** On completion or failure, a `ChatMessage::System` is posted to the main chat with: agent prompt (truncated), status, token usage, files modified (if any).

### 1.6 Commands

| Command | Action |
|---------|--------|
| `/bg <prompt>` | Spawn a background agent with the given prompt |
| `/bg --model <id> <prompt>` | Spawn with a specific model |
| `/bg list` | Show all background agents (running, complete, failed) with token usage |
| `/bg kill <id>` | Abort a running background agent |

### 1.7 Config

The existing `[background_agents]` TOML section (added in v0.7.1) is wired into `BackgroundAgentConfig`:

```toml
[background_agents]
max_agents = 3          # max concurrent (default: 5 in BackgroundAgentConfig)
per_agent_budget = 100000   # tokens per agent (default: 100_000)
global_budget = 500000      # total tokens across all agents (default: 500_000)
```

**Config mapping:** `BackgroundAgentSchemaConfig.max_agents` maps to `BackgroundAgentConfig.max_concurrent`. This mapping happens in `tui/src/app/state.rs` during `BackgroundAgentManager` initialization. Rename `max_concurrent` to `max_agents` in `BackgroundAgentConfig` to align the names and avoid confusion.

### 1.9 `spawn_background` Tool

Defined inline in `tui/src/agent/tools.rs` (same pattern as `spawn_agent`), not in the `ToolRegistry`. The tool definition:

- **name:** `spawn_background`
- **parameters:** `prompt` (required string), `model` (optional string)
- **execution:** handled at a higher level in `tool_handlers.rs` (before `execute_tool`), since it needs access to `State`, `Provider`, and `BackgroundAgentManager` â€” same dispatch pattern as `spawn_agent`

## Feature 2: Embedded SearXNG Setup

### 2.1 Overview

Caboose embeds the minimal Docker Compose configuration for SearXNG + Redis and manages the lifecycle. Zero user inputs required. No Cloudflare Tunnel â€” local only, architected so remote access is addable later via config.

### 2.2 Entry Points

1. **`/init`** â€” after generating CABOOSE.md, offers: "Set up web search? Requires Docker (y/n)". If yes, runs the setup flow.
2. **`/search-setup`** â€” standalone command for users who skipped init or want to set it up later.
3. **`/search-setup stop`** â€” runs `docker compose down` to tear down containers.
4. **`/search-setup status`** â€” checks if SearXNG containers are running and healthy.

### 2.3 Generated Files

Written to `~/.config/caboose/search/`:

**`docker-compose.yml`:**
```yaml
services:
  redis:
    image: redis:7-alpine
    container_name: caboose-redis
    restart: unless-stopped

  searxng:
    image: searxng/searxng:2026.3.10-8b95b2058
    container_name: caboose-searxng
    restart: unless-stopped
    ports:
      - "127.0.0.1:8080:8080"
    volumes:
      - ./searxng:/etc/searxng
    depends_on:
      - redis
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:8080/search?q=test&format=json"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

Key differences from tender-highball:
- Bound to `127.0.0.1:8080` (not `0.0.0.0`) â€” local only
- No Cloudflare Tunnel container
- Container names prefixed with `caboose-` to avoid collisions

**`searxng/settings.yml`:**
Generated with:
- Random secret key (32-byte hex, generated via `rand` crate)
- `base_url: "http://127.0.0.1:8080"`
- `public_instance: false`
- JSON format enabled
- Google + Bing engines (DuckDuckGo disabled by default due to CAPTCHA issues)
- Redis URL pointing to the companion container

### 2.4 Setup Flow

1. Check `docker compose` is available (run `docker compose version`).
2. If not found, post system message: "Docker is required for web search. Install it from docker.com"
3. Create `~/.config/caboose/search/` and `~/.config/caboose/search/searxng/` directories.
4. Generate secret key, write `docker-compose.yml` and `searxng/settings.yml`.
5. Run `docker compose up -d` from the search directory.
6. Poll `http://127.0.0.1:8080/search?q=test&format=json` until healthy (max 30s, 1s intervals).
7. On success: auto-write to global config (`~/.config/caboose/config.toml`):
   ```toml
   [services.web_search]
   provider = "searxng"
   base_url = "http://127.0.0.1:8080"
   ```
   **Config write strategy:** Read the existing file as a string. If `[services.web_search]` section already exists, skip the write and post "Web search config already present â€” not overwriting." Otherwise, append the TOML block to the end of the file. This avoids parse-reserialize (which would strip comments and reorder fields) while being safe against duplicates. Uses simple string search for `[services.web_search]`.
8. Post system message: "Web search enabled. SearXNG running on localhost:8080."
9. On timeout: post error with suggestion to check `docker compose logs -f caboose-searxng`.

### 2.5 Idempotency

If `~/.config/caboose/search/docker-compose.yml` already exists:
- `/search-setup` checks if containers are running. If yes, posts "Web search is already running." If not, starts them.
- Never overwrites existing Docker/settings files (preserves user customizations to settings.yml).
- Config write: skips if `[services.web_search]` already exists in config (see step 7 above).

### 2.6 Remote Access (Future)

The architecture supports adding remote access later without code changes:
- User adds a Cloudflare Tunnel container to the compose file manually
- Or changes `bind` from `127.0.0.1` to `0.0.0.0` and configures their own reverse proxy
- The `[services.web_search]` config already supports any `base_url`

### 2.7 Files to Create/Modify

- `core/src/search_setup.rs` (new) â€” compose file generation, settings generation, Docker health check, config auto-write
- `tui/src/app/slash_commands.rs` â€” add `/search-setup` command handler, modify `/init` to offer search setup
- `tui/src/tui/command.rs` â€” register `/search-setup` in command registry

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Docker not installed | "Docker is required for web search. Install it from docker.com" |
| Docker running but compose fails | Show compose stderr output |
| Health check timeout (30s) | "SearXNG failed to start. Check logs with: docker compose -f ~/.config/caboose/search/docker-compose.yml logs" |
| `/bg` with budget exhausted | "Background agent budget exceeded. Increase in [background_agents] config or wait for running agents to complete" |
| `/bg` with concurrency limit hit | "Maximum concurrent background agents reached ({n}/{max}). Wait for one to finish or use /bg kill" |
| `/bg kill` with invalid id | "No background agent with id '{id}'" |
| `/bg --model` with unknown model | "Unknown model '{id}'. Use /model to see available models" |
| SearXNG containers already running | "Web search is already running on localhost:8080" |

## Out of Scope

- `/bg attach` / `/bg detach` â€” deferred, revisit if users request it
- Cloudflare Tunnel integration â€” users who want remote access can modify compose files manually
- Auto-detecting when to background a task â€” the LLM decides via `spawn_background` tool; no heuristic-based auto-detection
- `/settings` UI for background agent config â€” TOML config is sufficient for now
- Background agents running across sessions â€” agents are session-scoped, killed on `/new` or quit
